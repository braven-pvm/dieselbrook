### 5. Pricing Architecture — B2B Debate Resolved
User asked about the history of Shopify B2B vs custom middleware debate for consultants.

**Conclusion**: B2B was early framing before SQL archaeology. It is NOT the correct path.

**Confirmed pricing model** (from `sp_camp_getSPprice` + `sp_ct_updatedisc` archaeology):
- Effective consultant price = `CampDetail.nPrice` if active campaign row exists, else `icitem.nprice`
- All branches of `sp_ct_updatedisc` resolve to flat 20% consultant discount
- Live pricing tables: `Campaign`, `CampDetail`, `CampSku`, `icitem`
- `soxitems`, `socamp`, `sosppr` = legacy lineage only

**Recommended architecture** (confirmed, in analysis/20 and analysis/21):
- Middleware precomputes effective consultant prices using `sp_camp_getSPprice` as oracle
- Sync as Shopify product metafields
- Deterministic Shopify Function applies `fixed_discount = retail - synced_consultant_price`
- Delta sync: `CampDetail.dLastUpdate` + `Campaign.dLastUpdate` + mandatory boundary sweep every 5–15 min (catches `dFrom`/`dTo` transitions without edits)

**Shopify Function availability constraint**:
- Custom app + Shopify Functions = **Shopify Plus only**
- Public App Store app = any plan
- App distribution model is the key variable — not B2B
