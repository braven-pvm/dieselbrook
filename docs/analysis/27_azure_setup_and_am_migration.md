### Phase 1 — Network Layer

**Do this first.** Everything else (VMs, App Services, SQL) is placed inside this network. Provisioning order matters — you can't put resources in a subnet that doesn't exist yet.

```bash
az group create --name rg-dbm-staging --location southafricanorth

az deployment group create \
  --resource-group rg-dbm-staging \
  --template-file infra/bicep/main.bicep \
  --parameters infra/bicep/parameters/staging.bicepparam \
  --what-if    # review first, then re-run without --what-if
```

**What gets created:**

| Resource | CIDR / Detail |
|---|---|
| `vnet-dbm-staging` | `10.1.0.0/16` |
| `snet-app` | `10.1.1.0/24` — App Services (VNet Integration delegation) |
| `snet-am` | `10.1.2.0/24` — AM IaaS VM. NSG: SQL (1433) from snet-app only. RDP from Bastion only. Deny-all inbound at 4000. |

| `snet-servicebus` | `10.1.5.0/24` — Service Bus private endpoint (prod only) |

Repeat for parity (`10.2.0.0/16`) and production (`10.3.0.0/16`).

### Phase 2 — Security Layer

Do this before provisioning compute. App Services reference Key Vault at startup — if Key Vault doesn't exist or secrets aren't loaded, the app fails to start.

**Key Vault provisioning (done by Bicep):**
- RBAC model (not legacy access policies)
- Soft delete enabled on all; purge protection enabled on production only
- Bicep creates the vault; we load secrets manually after

**Load secrets:**
```bash
az keyvault secret set --vault-name kv-dbm-staging --name am-sql-password       --value "<generated>"
az keyvault secret set --vault-name kv-dbm-staging --name dbm-state-db-password --value "<generated>"
az keyvault secret set --vault-name kv-dbm-staging --name shopify-api-secret     --value "<from partner dashboard>"
az keyvault secret set --vault-name kv-dbm-staging --name shopify-webhook-secret --value "<from partner dashboard>"
az keyvault secret set --vault-name kv-dbm-staging --name shopify-access-token   --value "<from app install>"
# servicebus-connection-string added after Service Bus is provisioned
```

**RBAC assignments (done by Bicep, verify manually):**

| Identity | Role | Scope |
|---|---|---|
| `app-dbm-core-staging` system identity | Key Vault Secrets User | `kv-dbm-staging` |
| Your Azure account | Key Vault Secrets Officer | `kv-dbm-staging` |
| `sp-dbm-github` service principal | Key Vault Secrets User | `kv-dbm-staging` |
| `app-dbm-core-staging` system identity | Azure Service Bus Data Owner | `sbns-dbm-staging` |
| `app-dbm-core-staging` system identity | Storage Blob Data Contributor | `stdbmstaging` |

App Settings in App Service use Key Vault reference syntax — never plain text secrets:
```
@Microsoft.KeyVault(VaultName=kv-dbm-staging;SecretName=am-sql-password)
```

### Phase 3 — AccountMate IaaS VM

**Provisioned by Bicep (`am-vm.bicep`):**

| Parameter | Staging | Production |
|---|---|---|
| VM SKU | `Standard_D4s_v5` (4 vCPU, 16 GB) | `Standard_D8s_v5` (8 vCPU, 32 GB) |
| OS | Windows Server 2022 Datacenter | Windows Server 2022 Datacenter |
| SQL Server | Developer edition (free) | Standard edition |
| OS disk | 128 GB Premium SSD | 256 GB Premium SSD |
| Data disk | 512 GB Premium SSD (D:\) | 1 TB Premium SSD |
| Public IP | None | None |
| Access | JIT VM Access | JIT VM Access |

**After VM is up — manual steps (via JIT RDP):**

**Step 3.1 — SQL Server configuration**

RDP in (request JIT access first via Azure Portal → VM → Connect → Request access). Run `scripts/am/configure-sql-server.ps1`:

- Sets max server memory to 12,288 MB (leaves 4 GB for OS)
- Enables mixed-mode authentication (Windows + SQL logins)
- Configures TempDB to 4 files (matches CPU count, reduces contention)
- Sets default data/log paths to D:\ (data disk, not OS disk)
- Enables backup compression
- Sets `COMPATIBILITY_LEVEL = 150` (SQL Server 2019 compatibility — conservative start)
- `OPTIMIZE FOR AD HOC WORKLOADS = ON` (reduces single-use plan cache bloat)

**Step 3.2 — Seed databases from production backup**

Annique IT takes a backup of production AM and uploads to Azure Blob:
```
stdbmstaging / am-backups /
  amanniquelive_20260408.bak
  compplanLive_20260408.bak
  compsys_20260408.bak
```

Run `scripts/am/seed-staging-am.ps1` on the VM. What it does:
1. Downloads `.bak` files from Blob Storage (Azure CLI, login auth)
2. `RESTORE DATABASE` for each — moves files to D:\SQLData\
3. `ALTER DATABASE SET COMPATIBILITY_LEVEL = 150` on all three
4. Creates `dbm_svc` SQL login with exact least-privilege grants:

| Database | Objects | Permission |
|---|---|---|
| `amanniquelive` | `icitem`, `arcust`, `SOPortal`, `soxitems`, `Campaign`, `CampDetail`, `CampSku`, `arinvc`, `soship`, `sostrs`, `sosord`, `soxitem`, `soskit`, `iciimgUpdateNOP` | SELECT |
| `amanniquelive` | `sp_camp_getSPprice`, `sp_ws_syncorder`, `sp_ws_reactivate`, `sp_ws_gensoxitems` | EXECUTE |
| `compplanLive` | `CTstatement`, `CTdownlineh`, `CTstatementh` | SELECT |
| `compsys` | `MAILMESSAGE` | SELECT |

5. `ALTER LOGIN sa DISABLE` — critical. No `sa` after seeding.

**Step 3.3 — Verify connectivity**

From a machine inside the VNet (or from the App Service via VNet Integration), run `scripts/am/verify-am-connection.ps1`:

```
[PASS] icitem read
[PASS] arcust read
[PASS] sp_camp_getSPprice
[PASS] sp_ws_gensoxitems
[PASS] CTstatement read
[PASS] MAILMESSAGE read

All checks passed. AM connectivity verified.
```

Do not proceed to Phase 5 until all checks pass.

## Part 2 — AccountMate Migration to Azure

### Why this is the blocker

DBM staging works today because we have a staging AM VM seeded from a production backup. That staging model **is** the production model — the production deployment is just that same VM pointed at the live traffic. The only remaining question is whether production AM lives on-prem (temporary, with VPN) or on Azure IaaS (permanent, preferred).

Given the team is non-technical and already struggling with the on-prem setup, the right call is to get AM onto Azure IaaS now — while we're building — rather than as a later migration. Staging has already proven the model works.

### Why this is actually straightforward

AccountMate is a traditional Windows client/server ERP. It uses ODBC to talk to SQL Server. There is no custom cloud connector, no containerisation complexity, no licensing model that changes in the cloud. It is, from a migration perspective, one of the simplest possible enterprise apps to lift and shift:

```
On-premises:          Azure IaaS:
  Windows Server    →   Windows Server 2022 (same OS, newer version)
  SQL Server        →   SQL Server 2022 (same engine, newer version)
  AM application    →   AM application (same installer, same config)
  ODBC DSN          →   ODBC DSN → localhost (SQL on same VM)
```

No schema changes. No data transformation. No API migration. Backup → upload → restore → install → verify.

### Phase C — Database Migration

This is identical to the staging seed process. The same scripts handle it.

**Full migration steps:**

1. **Pre-migration backup** (on-prem SQL Server):
```sql
BACKUP DATABASE amanniquelive TO DISK = 'C:\Backup\amanniquelive_migrate.bak' WITH COMPRESSION, STATS = 10;
BACKUP DATABASE amanniquenam   TO DISK = 'C:\Backup\amanniquenam_migrate.bak'  WITH COMPRESSION, STATS = 10;
BACKUP DATABASE compplanLive   TO DISK = 'C:\Backup\compplanLive_migrate.bak'  WITH COMPRESSION, STATS = 10;
BACKUP DATABASE compsys        TO DISK = 'C:\Backup\compsys_migrate.bak'       WITH COMPRESSION, STATS = 10;
```

2. **Upload to Azure Blob** (from on-prem machine with Azure CLI):
```bash
az storage blob upload \
  --account-name stdbmprod \
  --container-name am-backups \
  --name amanniquelive_migrate.bak \
  --file C:\Backup\amanniquelive_migrate.bak \
  --auth-mode login
```

3. **Restore on Azure VM** (run `seed-staging-am.ps1` adapted for production — same logic):
   - Downloads from Blob
   - `RESTORE DATABASE` with `WITH REPLACE, MOVE ...` to D:\SQLData\
   - Sets `COMPATIBILITY_LEVEL = 150`
   - Creates `dbm_svc` with exact grants
   - **Does NOT disable `sa` yet** — keep it active until AM application is verified working

4. **Verify AM application opens and data is correct:**
   - Open AM on the Azure VM itself (RDP session) — verify data loads, reports run, consultant lookups work
   - Spot-check a recent order in `amanniquelive`
   - Confirm `compplanLive` commission statements visible

5. **Disable `sa` after verification:**
```sql
ALTER LOGIN sa DISABLE;
```

**SQL Server version considerations:**

On-prem is SQL 2019 (compatibility level 150). Azure VM is SQL 2022. Restoring a SQL 2019 backup to SQL 2022 with `COMPATIBILITY_LEVEL = 150` is safe — SQL 2022 runs in 2019 compatibility mode without issues. Leave it at 150 initially. After AM is confirmed stable (2-4 weeks), upgrade to 160 (SQL 2022 native) for performance improvements. This is a one-line change: `ALTER DATABASE amanniquelive SET COMPATIBILITY_LEVEL = 160`.

### Phase D — Client Connectivity

This is the part that trips up non-technical teams. Clients currently connect to `AMSERVER-v9` on the local LAN. After migration, that server is in Azure. They need a way to reach it.

**Three options:**

#### Option 1 — Site-to-Site VPN (Recommended)

A VPN tunnel between the Annique office network and the Azure VNet. Once configured, every machine on the LAN can reach the Azure VM as if it's a local server.

```
Annique office LAN ←──VPN tunnel──→ Azure VNet (10.3.0.0/16)
  All workstations                     vm-am-prod (10.3.2.x)
  can reach 10.3.2.x directly
```

**What's needed:**
- Azure VPN Gateway provisioned in the DBM prod VNet (~$140/month Standard SKU)
- A VPN-capable router/firewall at the Annique office (FortiGate 40F/60F, Cisco, Meraki, pfSense — ~R3,000–R15,000 if they don't have one)
- One-time setup: configure IKEv2 tunnel with pre-shared key
- Client machines: update ODBC DSN from `172.19.16.100` to the Azure VM's private IP (e.g. `10.3.2.4`)

**How to update ODBC on Windows clients:**
```
Control Panel → Administrative Tools → ODBC Data Sources (64-bit)
→ System DSN → find "AccountMate" DSN
→ Edit → Server: change from 172.19.16.100 to 10.3.2.4
```
Or push via Group Policy if they have Active Directory.

**Best for:** offices with managed networking equipment. One-time setup. Transparent to end users after ODBC update.

#### Option 2 — Azure Virtual Desktop (AVD)

Run the AccountMate client application in the cloud. Users open a browser or the AVD app and get a desktop session running AM in Azure, next to the database.

```
User's laptop → browser/AVD app → Azure VM (running AM client) → vm-am-prod SQL Server
```

**What's needed:**
- Azure Virtual Desktop service (adds ~$50-100/month per user for compute)
- AM client installed on the AVD session hosts
- No ODBC changes on client machines (clients don't have AM installed locally)
- Users change how they launch AM: instead of clicking a desktop icon, they open AVD

**Best for:** when office networking is basic (home-grade routers, no IT to manage VPN). Higher cloud cost but zero client-side configuration.

#### Option 3 — Point-to-Site VPN

Individual client machines install a VPN certificate and connect on demand. Similar end result to S2S but without needing a VPN-capable office router.

**What's needed:**
- Azure VPN Gateway (same as S2S — ~$140/month)
- Certificate authority (Azure can generate one)
- Each client machine installs the certificate and the Azure VPN client app
- ODBC update on each client (same as S2S)

**Best for:** small number of clients, no managed office router. More per-machine setup than S2S.

**Summary recommendation:**

| Scenario | Recommended option |
|---|---|
| Annique has a managed router (FortiGate, Cisco, etc.) | Option 1 — S2S VPN |
| Annique has basic home-grade networking | Option 2 — Azure Virtual Desktop |
| < 5 AM users, any networking | Option 3 — P2S VPN |

**Action needed now:** Ask Annique IT what router/firewall brand is at the office. That single answer determines which option we use.

### Phase E — Cutover

This is the go-live switch. Minimal downtime if prepared correctly.

**Pre-cutover checklist:**
- [ ] Azure VM is provisioned and SQL Server is configured
- [ ] AM application server installed and licence transferred (not yet activated)
- [ ] Test restore completed and AM verified working in Azure (RDP session)
- [ ] VPN or AVD connectivity set up and tested with one client machine
- [ ] ODBC test: one client connects to Azure VM, opens AM, data visible
- [ ] DBM is running in production pointing at `vm-am-prod`
- [ ] Rollback plan documented: if anything goes wrong, point ODBC back to `172.19.16.100`

**Cutover sequence (aim for a quiet period — Friday evening or weekend):**

```
T-0: Announce freeze — no AM transactions from this point
T+5min: Final production backup (all 4 databases)
T+10min: Upload to Azure Blob
T+20min: Restore to vm-am-prod (scripts already tested — this is fast)
T+30min: Verify data in Azure VM — open AM, check last order matches
T+35min: Activate AccountMate licence on Azure VM (call AM support or use portal)
T+40min: Update ODBC on client machines (or push via Group Policy)
T+45min: Test: open AM on a client machine → connect → create a test transaction
T+50min: Confirm test transaction visible in SQL on Azure VM
T+55min: Open to all users — cutover complete
T+ongoing: Monitor for 24 hours. Keep on-prem SQL Server running (read-only) for 1 week as fallback.
```

Expected downtime: **30–60 minutes**. Could be done during lunch or after hours.

---

### Rollback Plan

If anything fails after cutover:
1. Update ODBC on client machines back to `172.19.16.100` (on-prem)
2. On-prem SQL Server was left running — it has all data up to the freeze point
3. Any transactions entered into Azure AM during the failed cutover window need to be re-entered (keep the window short)

Keep on-prem SQL Server running for at least 1 week after successful cutover before decommissioning.
